# SolExplorer API

Backend-сервис для анализа и скоринга токенов в сети Solana. Принимает Mint Address токена, собирает данные о торговых парах с DEX-агрегатора, рассчитывает score надёжности и возвращает структурированный ответ.

Frontend: [token-explorer-front](https://github.com/nikkol615/token-explorer-front)

---

## Запуск

### Требования

- Python ≥ 3.12
- [Poetry](https://python-poetry.org/) для управления зависимостями

### Установка и запуск

```bash
# Клонировать репозиторий и перейти в директорию
cd token-explorer-back

# Установить зависимости
poetry install

# Запустить dev-сервер
poetry run uvicorn solexplorer.main:app --reload --port 8000
```

Сервер запустится на `http://localhost:8000`. Интерактивная документация доступна по адресам:
- Swagger UI: `http://localhost:8000/docs`
- ReDoc: `http://localhost:8000/redoc`

### Переменные окружения (опционально)

Создайте файл `.env` в корне проекта, если нужно переопределить настройки по умолчанию:

```env
DEXSCREENER_BASE_URL=https://api.dexscreener.com
HTTP_TIMEOUT=10.0
```

```
solexplorer/
├── main.py                    # Инициализация FastAPI, CORS, подключение роутеров
├── config/
│   └── app.py                 # Pydantic Settings (base URL, timeout, метаданные)
├── api/v1/
│   ├── router.py              # Корневой роутер /api/v1, объединяет sub-роуты
│   └── token_analyse.py       # Эндпоинт анализа токена
├── solana/
│   └── get_token.py           # Валидация адреса, запрос к DexScreener, агрегация
├── analyse/
│   └── get_score.py           # Движок скоринга (5 критериев, 0–100 баллов)
└── schemas/
    ├── token.py               # Pydantic-модели: DexPair, TokenOverview и т.д.
    └── score.py               # Pydantic-модели: TokenScore, ScoreBreakdownItem
```

**Принцип модульности:** роутеры не содержат бизнес-логики — они только принимают запрос, делегируют работу сервисным модулям (`solana/`, `analyse/`) и возвращают типизированный ответ.

### API Endpoint

У сервиса один эндпоинт, который делает всё:

```
GET /api/v1/token/{token_address}
```

| Параметр | Тип | Описание |
|----------|-----|----------|
| `token_address` | `string` (path) | Mint Address токена в сети Solana |

**Успешный ответ (200)** — `TokenAnalysisResponse`:

```json
{
  "overview": {
    "address": "So11111111111111111111111111111111111111112",
```

**Ошибки:**

| Код | Когда | Пример `detail` |
|-----|-------|-----------------|
| 400 | Невалидный Solana-адрес | `"Invalid Solana address: ..."` |
| 404 | Токен не найден / нет пулов | `"Token not found or has no liquidity pools"` |
| 503 | DexScreener недоступен | `"External API unavailable (timeout)"` |

## 2. Как получаем информацию о токенах

Весь процесс получения данных реализован в `solana/get_token.py` и состоит из трёх этапов:

### Шаг 1: Валидация адреса

Входящий адрес проверяется через библиотеку `solders` — это Rust-обёртка для работы с типами Solana. Вызов `Pubkey.from_string(address)` проверяет, что строка является корректным Base58-кодированным публичным ключом длиной 32 байта. Если адрес невалиден — сразу 400 ошибка, запрос к внешнему API не отправляется.

### Шаг 2: Запрос к DexScreener

Отправляем асинхронный GET-запрос через `httpx`:

```
GET https://api.dexscreener.com/latest/dex/tokens/{address}
```

DexScreener — бесплатный агрегатор DEX-данных, который индексирует торговые пары с десятков блокчейнов. В ответ приходит JSON со списком всех пар, в которых участвует данный токен (на любых чейнах). Мы парсим ответ в Pydantic-модель `DexScreenerResponse` и фильтруем только пары с `chainId == "solana"`.

Если DexScreener не ответил за 10 секунд или вернул HTTP-ошибку — возвращаем 503. Если список пар пуст — 404.

### Шаг 3: Агрегация в TokenOverview

Из отфильтрованного списка Solana-пар собираем сводку:

- **Идентификация токена:** берём первую пару, определяем — наш токен в ней `baseToken` или `quoteToken` (сравниваем адреса). Из нужной стороны извлекаем `name`, `symbol`, `address`.
- **Цена:** `priceUsd` из первой пары (приходит строкой от DexScreener, конвертируем в `float`).
- **Агрегированные метрики:** суммируем `liquidity.usd` и `volume.h24` по всем Solana-парам — получаем общую ликвидность и суточный объём.
- **DEX-листинг:** собираем уникальные `dexId` (Raydium, Orca, Meteora и т.д.).
- **FDV / Market Cap:** берём первое ненулевое значение из списка пар.
- **Изображение:** `imageUrl` из метаданных первой пары (если есть).

Полный список пар (`pairs`) также включается в ответ — фронтенд использует их для отображения таблицы пулов.

## 3. Как работает скоринг

Скоринг — это количественная оценка надёжности и зрелости токена по шкале от 0 до 100. Цель — дать пользователю быструю визуальную сигнализацию: стоит ли присмотреться к токену (🟢), действовать с осторожностью (🟡) или лучше обойти стороной (🔴).

Движок реализован в `analyse/get_score.py` и оценивает **пять независимых критериев**. Каждый критерий отражает конкретный риск-фактор.

### Критерий 1: Количество DEX (макс. 20 баллов)

| Условие | Баллы |
|---------|-------|
| 0 DEX | 0 |
| 1 DEX | 10 |
| ≥ 2 DEX | 20 |

**Зачем:** если токен торгуется только на одном DEX — это повышенный риск. Единственный пул может быть создан самим автором токена без реального интереса рынка. Присутствие на нескольких DEX (Raydium, Orca, Meteora) говорит о том, что разные маркет-мейкеры и пользователи создавали пулы независимо, что свидетельствует о реальном спросе.

### Критерий 2: Общая ликвидность (макс. 30 баллов)

| Условие | Баллы |
|---------|-------|
| ≤ $10,000 | 0 |
| $10,001 – $50,000 | 10 |
| $50,001 – $200,000 | 20 |
| > $200,000 | 30 |

**Зачем:** ликвидность — самый весомый критерий (30 баллов из 100). Токен с низкой ликвидностью подвержен манипуляциям: крупная покупка или продажа может сдвинуть цену на десятки процентов. При ликвидности меньше $10k любая сделка на $500–1000 вызовет значительный slippage. Токены с ликвидностью > $200k обеспечивают стабильную торговлю и сложнее поддаются pump-and-dump схемам.

### Критерий 3: Соотношение объёма к ликвидности (макс. 20 баллов)

| Условие | Баллы |
|---------|-------|
| Нет ликвидности | 0 |
| Ratio < 0.1 (слишком низкий) | 5 |
| 0.1 ≤ Ratio ≤ 5.0 (здоровый) | 20 |
| Ratio > 5.0 (слишком высокий) | 5 |

Формула: `Volume24h / TotalLiquidity`

**Зачем:** этот индикатор выявляет две крайности.
- **Ratio < 0.1** — «мёртвый» токен: ликвидность залита, но никто не торгует. Часто это заброшенные проекты или ловушки (honeypot), где ликвидность заблокирована, но сделки не проходят.
- **Ratio > 5.0** — подозрение на wash trading. Когда суточный объём в 5+ раз превышает ликвидность, это может означать, что одни и те же участники гоняют средства по кругу, создавая иллюзию активной торговли. Нормальный диапазон для здорового токена — от 0.1 до 5.0.

### Критерий 4: Концентрация ликвидности (макс. 20 баллов)

| Доля крупнейшего пула | Баллы |
|----------------------|-------|
| < 80% | 20 |
| 80% – 95% | 10 |
| ≥ 95% | 5 |

Формула: `MaxPoolLiquidity / TotalLiquidity`

**Зачем:** если > 95% всей ликвидности сосредоточено в одном пуле — это single point of failure. Создатель этого пула может вывести ликвидность (rug pull), и токен мгновенно потеряет возможность торговли. Распределённая ликвидность (< 80% в топ-пуле) означает, что несколько независимых провайдеров обеспечивают ликвидность, что делает токен устойчивее.

### Критерий 5: Возраст пулов (макс. 10 баллов)

| Возраст старейшего пула | Баллы |
|------------------------|-------|
| < 1 часа | 0 |
| 1 – 24 часа | 5 |
| > 24 часов | 10 |

Рассчитывается из поля `pairCreatedAt` (unix timestamp в миллисекундах от DexScreener). Берётся **самый старый** пул.

**Зачем:** свежесозданные токены — это наибольший риск. Подавляющее большинство скам-токенов на Solana существуют менее суток: создатель запускает токен, накачивает цену, выводит ликвидность и исчезает. Токен, пулы которого существуют больше 24 часов, уже прошёл начальный период наибольшего риска. Этот критерий имеет наименьший вес (10 баллов), потому что возраст сам по себе не гарантирует надёжность — но молодость точно увеличивает риск.

### Итоговая оценка

Баллы суммируются (максимум 100):

| Диапазон | Эмодзи | Интерпретация |
|----------|--------|---------------|
| 71–100 | 🟢 | Относительно надёжный токен: хорошая ликвидность, мультидекс, адекватные объёмы |
| 40–70 | 🟡 | Средний риск: возможно стоит изучить подробнее перед принятием решений |
| 0–39 | 🔴 | Высокий риск: низкая ликвидность, один DEX, подозрительные метрики или свежий токен |

Ответ всегда содержит `breakdown` — массив из 5 элементов с подробным описанием (`reason`) каждой оценки, чтобы пользователь видел не просто число, а понимал *почему* токен получил такой score.
